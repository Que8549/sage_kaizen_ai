## Architecture Risks Identified

- **[HIGH] Voice Startup Decoupling** (`src/voice_pipeline.py:202`)
  The `voice_ready` signal is now sent *before* the greeting audio plays (removed `drain()` wait). While this reduces startup latency, it introduces a dependency on the Main App to handle the greeting playback. If the Main App's "Ready" logic does not strictly wait for the `voice_ready` ZMQ message before triggering the greeting, the user might hear the greeting *before* the Voice Pipeline is fully initialized, potentially causing a slight audio glitch or timing mismatch.

- **[HIGH] Session State Persistence** (`src/_zmq_handlers.py:443`)
  The line `session_id = None` is intended to clear the active session after a turn. However, if `session_id` is a local variable passed as an argument to `run_tts_subscriber`, this assignment has no effect on the caller's state. If `session_id` is a module-level or class-level variable, the fix is correct. This ambiguity risks the "spurious barge-in" bug returning if the variable scope is not shared.
